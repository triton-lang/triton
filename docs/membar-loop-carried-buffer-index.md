# Loop-carried buffer indices in membar

**Temporary review note:** This document is informational only and exists to
help clarify the PR review discussion. It should be deleted before the PR is
merged.

This note explains why `BufferIndexAnalysis` invalidates the dynamic
buffer-index payload of slices that cross a loop backedge, and why non-loop
CFG merges don't need the same treatment. It describes the CFG form seen by
membar after `scf-to-cf` lowering.

## 1. Problem

### 1.1 How membar models loop dependencies

Membar runs a forward dataflow over the CFG
(`lib/Analysis/Membar.cpp:140-181`). For each (virtual) block it computes an
output `BlockInfo` (read/write slices) by running a transfer function over
the block's ops, then joins that output into the input of every successor
(`Membar.cpp:174`). The worklist re-enqueues successors until no input
changes.

The CFG below is the post `scf-to-cf` shape of any loop:

```
       ^bb_entry              in[entry]  = {}
           |
           v
       ^bb_header <-----+     in[header] = in[entry-edge] join in[backedge]
           |            |                            ^^^^^^^^^^^^^^^^^^^^^
           v            |                            previous iteration's
       ^bb_body --------+     in[body]   = in[header]   body output, looped
           |  (backedge)                              back via Membar.cpp:174
           v
       ^bb_exit              in[exit]   = in[body-exit-edge]
```

Because the backedge feeds `^bb_body`'s output back into `^bb_header`'s
input, after the dataflow reaches a fixed point the body's input contains
slices that were generated by *the previous iteration's body*. Concretely:

```
   loop iter 0:  in[header] = {}
                 out[body]  = { W, R }      <-- this iteration's slices

   loop iter 1:  in[header] = in[entry] join out[body]
                            = { W, R }      <-- carried in over backedge
                 in[body]   = { W, R }
                 out[body]  = { W, R } join { W, R } = { W, R }
                                                               (fixed point)
```

The `{ W, R }` state is a summary of memory operations that may need to be
ordered against later operations. It is not saying the previous iteration is
executed again; it means the next walk of the loop body checks current
operations against the summary carried through the loop header.

So during loop iter 1 the body's transfer function sees both:

- slices generated by the *current* iteration (about to be added), and
- slices generated by the *previous* iteration, carried back over the
  backedge into the header (already in `in[body]`).

This carried state lets membar detect cross-iteration RAW/WAR hazards with a
single in-order walk. It is also what makes the buffer-index disjointness
proof invalid for backedge-carried slices &mdash; see 1.2.

### 1.2 Why the proof is invalid across a backedge

`BufferIndexAnalysis` proves disjointness via an SSA-identity proof: two
slices with the same SSA base value and different constant offsets (mod the
same modulus) target different slots.

For a carried slice, the SSA value attached to the index is still the loop
header's block argument (e.g. `%phase`). But the carried slice was generated
when `%phase` denoted the *previous* iteration's runtime integer, while the
current iteration's slice has it bound to the *new* iteration's integer.
The same SSA value no longer implies the same runtime integer across the
backedge.

```
   loop header block argument %phase

       iter i-1:  %phase = 0        <-- carried slice captured here
       iter i  :  %phase = 1        <-- current slice captured here

       both refer to the same SSA Value, but to different runtime ints
```

## 2. Example: false disjointness across a backedge

Two-buffer pipeline (`N = 2`), writer at `(phase + 1) % N`, reader at
`phase % N`. After `scf-to-cf`:

### 2.1 IR

```mlir
^bb_body(%phase: i32):
  // write at (phase + 1) % N
  %wi = arith.addi %phase, %c1
  %w  = arith.remsi %wi, %cN
  %ws = ttg.memdesc_index %buf[%w]
  ttg.local_store %v, %ws

  // read at phase % N
  %r  = arith.remsi %phase, %cN
  %rs = ttg.memdesc_index %buf[%r]
  %x  = ttg.local_load %rs

  %phase_next = arith.addi %phase, %c1
  cf.br ^bb_body(%phase_next : i32)
```

### 2.2 Concrete slot timeline: barrier required

Per-iteration slot assignment:

```
   iter 0:  phase=0   write -> slot 1   read -> slot 0
   iter 1:  phase=1   write -> slot 0   read -> slot 1
   iter 2:  phase=2   write -> slot 1   read -> slot 0    (period 2)
```

Per-slot timeline:

```
   slot 0:   iter0:read  ->  iter1:write  ->  iter2:read  -> ...
                            ^^^^^^^^^^^^                  ^^^^^^^^^^^^
                              WAR (across backedge)         WAR

   slot 1:   iter0:write ->  iter1:read   ->  iter2:write -> ...
                            ^^^^^^^^^^^^                  ^^^^^^^^^^^^
                              RAW (across backedge)         RAW
```

These are real cross-iteration dependencies. In particular, `slot 1` has a
RAW dependency from the write in iter 0 to the read in iter 1. A barrier is
therefore required between adjacent iterations for this slot, even though the
write and read indices are different within a single iteration.

### 2.3 Without invalidation: barrier incorrectly skipped

Slices at the entry of `^bb_body` after the dataflow reaches a fixed point:

| origin (previous iter) | slice                       | `bufferIndexExpr`        |
|------------------------|-----------------------------|--------------------------|
| carried write          | `(phase + 1) % N` on `%buf` | `(%phase, +1, mod N)`    |
| carried read           | `phase % N`       on `%buf` | `(%phase,  0, mod N)`    |

When the body's *current* read at `phase % N` is processed, `intersects`
checks it against the carried *write*:

```
   carried write.bufferIndexExpr = (%phase, +1, mod N)
   current read.bufferIndexExpr  = (%phase, +0, mod N)

   areBufferIndicesProvablyDifferent  -> TRUE   (same base, 0 != 1 mod N)
   -> barrier incorrectly skipped
```

"Same SSA base" is false at runtime: the carried `%phase` is iter `i-1`'s
integer, the current `%phase` is iter `i`'s integer. The skipped barrier
misses the real cross-iteration RAW shown in 2.2.

### 2.4 With invalidation: barrier inserted

After the join at `Membar.cpp:174`, if the edge is a backedge
(`Membar.cpp:177-178`), `invalidateBufferIndices` clears `bufferIndexExpr`
on every carried slice:

| origin (previous iter) | slice                       | `bufferIndexExpr` |
|------------------------|-----------------------------|-------------------|
| carried write          | `(phase + 1) % N`           | `nullptr`         |
| carried read           | `phase % N`                 | `nullptr`         |

`areBufferIndicesProvablyDifferent` returns false on `nullptr`, so the
intersection check falls back to plain physical-interval aliasing on the
shared `%buf` allocation. The required barrier is inserted.

## 3. Why this is loop-specific

### 3.1 Non-loop CFG merges don't need invalidation

```
       if / else merge                         loop header

       ^bb_then    ^bb_else                    ^bb_header <--+
             \      /                               |        |
              v    v                                v        | backedge
             ^bb_merge                          ^bb_body     |
                                                    |        |
       one predecessor runs                        +--------+
       per dynamic execution;
       %v denotes the selected              header's block
       incoming value at the              argument is re-bound
       join                               on every iteration
```

At a non-loop merge, the merge block argument may select different incoming
values from different predecessors. The key point is that, within one
dynamic execution of the merge block, all uses of that block argument denote
the selected value for that same execution. The analysis compares slices
from that same execution, so the SSA-identity proof stays valid &mdash; no
invalidation needed.

A loop backedge is the case relevant to this analysis where a block argument
is re-bound to a *different* iteration's runtime values, so it is where the
proof no longer applies.

### 3.2 Detecting a backedge

Standard dominance rule: an edge `pred -> succ` is a backedge iff `succ`
dominates `pred`. This is exactly what `BufferIndexAnalysis::
isBackedgeSuccessor` checks, called at `Membar.cpp:177` and using the
per-function `DominanceInfo` owned by `BufferIndexAnalysis`.

### 3.3 Conservative fallback

We currently discard the index payload entirely on backedges, so we can't
prove cross-iteration disjointness even when the modular arithmetic would
allow it (e.g. `N >= 3`). A backedge index rewrite that advances the carried
offset by the per-iteration delta of the block argument (mod `N`) could
recover this precision without changing the membar interface; it would
replace invalidation with a per-slice rewrite. This PR keeps the backedge
behavior conservative.
