# IR Comparison: Before and After TritonGPUPipeline

## Confirmed Facts (from actual `triton-opt` output)

### Method

The before/after IR was generated by running `triton-opt` with the actual Triton
pipelining passes on a hand-crafted MLIR input representing the `gemm_scatter_kernel`.

```bash
# Before pipeline (after assign-latencies + schedule-loops):
triton-opt input.mlir -tritongpu-assign-latencies -tritongpu-schedule-loops -canonicalize

# After pipeline (full sequence):
triton-opt input.mlir -tritongpu-assign-latencies -tritongpu-schedule-loops -tritongpu-pipeline=num-stages=3 -canonicalize
```

  - `tt.load %arg7` (A tile load) ÔÇö has no latency attribute yet in ScheduleLoops output
  - `ttg.convert_layout` with `{loop.stage = 2, loop.cluster = 0}`

  - `ttg.convert_layout` with `{loop.stage = 2, loop.cluster = 0}`
  - `arith.mulf` with `{loop.stage = 2, loop.cluster = 0}` (b_scale multiplication)

  - `scf.yield`

### After Pipeline: `artifacts/ir_after/after_pipeline.mlir`

Structure (92 lines total, ~2.5x larger):
- Lines 1-6: Layout definitions ÔÇö added `#shared`, `#shared1`, `#smem` (shared memory types)
- Lines 9-13: New constants: `%c2`, `%c2_i32`, `%c1_i32`, `%c0_i32`, `%c-1_i32` (buffer index tracking)
- Lines 18-19: New constants: `%cst_3`, `%cst_4` ÔÇö zero tensors for masked async copy "other" values
- Lines 30-31: **`ttg.local_alloc` ├ù 2** ÔÇö allocates double-buffered shared memory:
  - `!ttg.memdesc<2x128x32xf16, #shared, #smem, mutable>` (2 buffers for A)
  - `!ttg.memdesc<2x32x128xf16, #shared1, #smem, mutable>` (2 buffers for B)

**Prologue (lines 32-52):** First 2 iterations unrolled:
- Iteration 0:
  - `ttg.memdesc_index %10[%c0_i32]` ÔÇö select buffer 0
  - `ttg.async_copy_global_to_local %4, %13 mask ...` ÔÇö async copy A into shared mem
  - `ttg.async_commit_group tokens %15` ÔÇö commit copy group
  - Same for B tile
- Iteration 1:
  - `tt.addptr` to advance pointers
  - `ttg.memdesc_index %10[%c1_i32]` ÔÇö select buffer 1
  - `ttg.async_copy_global_to_local %23, %25` ÔÇö async copy A ahead
  - Same for B tile

**Main loop (lines 53-81):** Pipelined `scf.for` with 9 iter_args (was 3):
- Added iter_args: `%arg10` (ins_idx), `%arg11` (ext_idx), plus 4 `!ttg.async.token` values
- Loop body:
  - **Consume:** `ttg.async_wait %arg12, %arg14 {num = 2}` ÔÇö wait for oldest copies
  - `ttg.memdesc_index` ÔåÆ `ttg.local_load` (load from shared memory into registers)
  - `arith.mulf` + `tt.dot` ÔÇö compute (unchanged structure)
  - **Produce:** `ttg.async_copy_global_to_local` ├ù 2 ÔÇö issue new async copies
  - `ttg.async_commit_group` ÔÇö commit new group
  - Buffer index modular arithmetic (`arith.addi` + `arith.cmpi` + `arith.select`)

**Epilogue (lines 82-84):**
- `ttg.async_wait {num = 0}` ÔÇö drain all pending async copies
- `ttg.local_dealloc` ├ù 2 ÔÇö free shared memory buffers

**Post-loop scatter store (lines 85-87):** **UNCHANGED:**
```mlir
    %35 = ttg.convert_layout %33#2 : tensor<128x128xf32, #mma> -> tensor<128x128xf32, #blocked1>
    %36 = tt.splat %arg5 : !tt.ptr<f32> -> tensor<128x128x!tt.ptr<f32>, #blocked1>
    tt.store %36, %35 : tensor<128x128x!tt.ptr<f32>, #blocked1>
```

This is **identical** to the before-pipeline version (modulo SSA value renaming: `%12ÔåÆ%36`, `%11ÔåÆ%35`, `%10#2ÔåÆ%33#2`). The pipeliner did not modify, move, or annotate this store in any way.

## Observed Behavior

1. The pipeliner replaced `tt.load` with `ttg.async_copy_global_to_local` + shared memory buffering
2. It added prologue unrolling (2 iterations ahead) and epilogue (drain pending copies)
3. The `tt.dot` op itself is unchanged ÔÇö it now reads from `ttg.local_load` (shared mem) instead of `ttg.convert_layout` (global load result)
4. **The `tt.store` to the output pointer is completely untouched** ÔÇö same structure, same position, same semantics
